\begin{abstract}
This report evaluates Version 2 of the Slovenian tax RAG pipeline using the JSONL logs from SLURM job \texttt{14625503}. Version 2 replaces the original fixed-only retrieval experiment with article-aware legal chunking and hybrid retrieval. The improved retriever raises source hit@3 from 40.0\% to 95.0\%, article hit@3 from 0.0\% to 30.0\%, and exact phrase hit@3 from 0.0\% to 35.0\%. Generation remains the main limitation: strict prompting is slightly more correct than the baseline prompt, but manual correctness is still only 0.65/2 on average because many failures originate from missing exact articles in the retrieved context.
\end{abstract}
\section{Version 2 Pipeline}
The system is a Retrieval-Augmented Generation pipeline for Slovenian tax law. It ingests local PISRS HTML files, extracts visible text, chunks the documents, embeds chunks with a multilingual sentence-transformer, stores vectors in FAISS, retrieves the most similar chunks for a question, and passes those chunks to a local instruction-tuned language model.
\subsection{Document Processing and Legal Chunking}
Version 1 used only overlapping fixed-size character chunks. Version 2 adds an article-aware chunking strategy in \texttt{src/chunking.py}. The chunker detects legal headings such as \texttt{3. člen} and \texttt{113.a člen}, splits documents into legal sections, and attaches metadata: source filename, law identifier, document role, article number, article title, and section position. Long articles are still split into overlapping subchunks, but continuation chunks are prefixed with the law and article metadata so that embeddings retain the legal reference.
\subsection{Embedding and Similarity Search}
The embedding model is \texttt{sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2}. During indexing, every chunk text is mapped to a dense vector and normalized. FAISS \texttt{IndexFlatIP} stores the vectors. Because vectors are normalized, inner product ranking is equivalent to cosine similarity. For a question $q$, the system embeds $q$ into the same vector space and retrieves dense candidates by maximum cosine similarity.
\subsection{Hybrid Reranking}
The Version 2 retriever first asks FAISS for a larger candidate set \texttt{candidate\_k=100} and then reranks candidates with a hybrid score:
\[
s(c, q) = s_{\mathrm{dense}}(c, q) + 0.25 \cdot s_{\mathrm{lexical}}(c, q) + s_{\mathrm{law}}(c, q)
\]
where $s_{\mathrm{lexical}}$ is normalized token overlap between the question and chunk text plus metadata, and $s_{\mathrm{law}}$ is a 0.20 boost when the question names a law such as ZDDV-1, ZDoh-2, ZDDPO-2, or ZDavP-2 and the chunk metadata matches that law. The final prompt uses \texttt{top\_k=3} chunks.
\subsection{Generation}
The generation step formats retrieved chunks with source filename, chunk id, score, law id, article number, and article title. Two prompt variants completed in the available logs: the original baseline prompt and a stricter prompt. The strict prompt explicitly tells the model to prefer the chunk whose legal act and article directly answer the question and to say that the answer was not found if the exact rule is missing. The extractive prompt was configured in the SLURM script but no extractive JSONL file is present for job \texttt{14625503}, so it is excluded from this report.
\section{Evaluation Design}
The evaluation set contains 20 manually written tax questions. Each case includes expected source files, expected article locations, previous expected chunk IDs, key phrases, and a reference answer. Because Version 2 changes chunking, exact expected chunk IDs from Version 1 are no longer a stable primary metric. Therefore this report emphasizes source hit, article hit, phrase hit, manual context relevance, faithfulness, and correctness.
\begin{itemize}[leftmargin=*]
\item \metric{Context relevance} (0--2): whether retrieved chunks contain the exact legal rule or only related material.
\item \metric{Faithfulness} (0--2): whether the generated answer is supported by the retrieved chunks.
\item \metric{Answer correctness} (0--2): whether the generated answer matches the expected legal answer.
\end{itemize}

\paragraph{Manual generation assessment.}
The second-pass manual assessment gives the baseline prompt an average faithfulness of 1.55/2 and correctness of 0.60/2. The strict prompt gives faithfulness 1.55/2 and correctness 0.65/2. The strict prompt is slightly better on correctness, mainly because it is more likely to refuse or narrow the answer when context is incomplete. It does not solve retrieval failures.
\paragraph{Chunk comparison.}
The retrieved top-3 chunk sets are identical between the retrieval-only, baseline-prompt, and strict-prompt legal-hybrid runs for 20/20 cases. This means the prompt comparison is controlled: the model saw the same context in both prompt variants. The legal-hybrid top-3 chunks match the fixed dense top-3 chunks in 0/20 cases, showing that Version 2 materially changed retrieval. Exact expected chunk hit@3 remains 0/20 because expected chunk IDs were written for the previous chunking scheme; article and phrase hits are more meaningful for Version 2.

\section{Main Findings}
\begin{enumerate}[leftmargin=*]
\item Article-aware chunking improves metadata quality and makes evaluation more meaningful. Most retrieved chunks now carry law and article information.
\item Hybrid retrieval greatly improves source-level retrieval, but it still often retrieves a related article instead of the exact definition article. This is the dominant error source.
\item Exact chunk ID matching is no longer a useful success criterion across pipeline versions, because legal chunking creates different chunk IDs. Source, article, and expected phrase coverage should be used instead.
\item Generation is usually faithful to retrieved context, including wrong context. This is useful diagnostically: answer failures mostly follow retrieval failures, not arbitrary hallucination.
\item Strict prompting improves concise behavior and some correctness cases, but cannot repair missing context. It sometimes answers in English even for Slovenian questions, so language control should be tightened.
\end{enumerate}
\section{Recommendations for the Next Iteration}
\begin{itemize}[leftmargin=*]
\item Add explicit article-number matching when the question contains or implies a known article, and include legal synonyms such as predmet obdavčitve, davčni zavezanec, davčna osnova, and stopnja davka.
\item Add BM25 or another sparse retriever and combine it with dense retrieval. Current lexical overlap helps but is too weak for exact legal definitions.
\item Index article titles separately and boost title matches. Many evaluation questions ask for the article title concept directly.
\item Rebuild the evaluation expected chunks for the legal chunking scheme, or replace chunk-id expectations with source+article+phrase checks.
\item Keep strict prompting as the default generation prompt, but add an explicit instruction to answer in Slovenian for Slovenian questions and to avoid citing non-answer chunks.
\end{itemize}
